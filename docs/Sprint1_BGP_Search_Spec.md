# Sprint 1 - BGP Intelligent Search Specification

## Purpose

The purpose of the BGP Intelligent Search feature is to identify all BGP-related configuration on a network device and provide the complete operational context required to understand how BGP is configured and how BGP sessions operate.

The search should not only display BGP configuration statements but should also identify related configuration objects that influence BGP operation.

The output should help a network engineer troubleshoot BGP issues by showing:

- The BGP configuration itself.
- The supporting configuration referenced by BGP.
- The relationship between BGP and related configuration objects.

---

## Search Input

The user provides:

Technology:

BGP

Example:

Search:
BGP

The search engine should identify:

- BGP process configuration.
- BGP neighbors.
- BGP policies.
- Referenced configuration objects.
- Related context configuration.

---

## Output

The output is divided into two sections:

### 1. BGP Configuration

This section displays the configuration that directly defines the BGP process.

The output should include:

### BGP Process

- Local Autonomous System Number (ASN)
- BGP Router ID
- Address Families
- VRF association (if applicable)

### BGP Neighbors

For each BGP neighbor, display:

- Neighbor IP address
- Remote ASN
- Neighbor description
- Update source
- Authentication configuration
- BFD configuration
- Peer group association
- Route policies applied
- Prefix filtering applied


### BGP Advertisement Configuration

Display:

- Network statements
- Aggregate addresses
- Redistribution configuration
- Default route advertisement


### BGP Policy Configuration

Display:

- Route-map references
- Prefix-list references
- Community policy references
- AS path filtering references


---

### 2. Context Configuration

This section displays configuration outside the BGP process that is required to understand BGP operation.

Context configuration should be identified by following references from BGP configuration.

The context should include:

---

### Interfaces

Display interfaces referenced by BGP.

Examples:

- Update source interfaces
- Interfaces used for BGP reachability

Include:

- Interface name
- IP address
- VRF association
- Interface status (if available)


---

### VRF Configuration

Display VRFs associated with BGP.

Include:

- VRF name
- Route Distinguisher (if applicable)
- Route Targets (if applicable)


---

### Route Maps

Display route maps referenced by:

- Neighbor statements
- Redistribution
- Policy configuration

Include:

- Route map name
- Match conditions
- Set actions
- Referenced objects


---

### Prefix Lists

Display prefix lists referenced by:

- Neighbor filtering
- Route maps

Include:

- Prefix list name
- Permitted/denied prefixes


---

### Access Control Lists

Display ACLs referenced by:

- Route maps
- Neighbor filtering

Include:

- ACL name
- Matching conditions


---

### Other Context Objects (Future Expansion)

The following objects may be included in future versions:

- Community Lists
- AS Path Access Lists
- BFD Configuration
- Routing Protocol Context (OSPF, IS-IS, Static Routes)
- MPLS VPN Context
- Peer Templates


---

## Notes

### Design Principles

1. NetSage should understand networking concepts rather than vendor-specific commands.

2. The BGP search engine should be vendor-neutral.

3. Vendor-specific configuration should be converted into common network objects.

4. BGP configuration and Context Configuration should remain separate.

5. Context Configuration should be generated by following references between configuration objects.

Example:

BGP Neighbor
|
references
|
Route Map
|
references
|
Prefix List

6. The first implementation will focus on BGP only.

7. The initial parser implementation may support one vendor, but the underlying design should allow future multi-vendor support.
